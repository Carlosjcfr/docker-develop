# Análisis: Refactorización y División de la Librería Core (`lib.sh`)

## 📌 Contexto de la Propuesta
Actualmente toda la lógica transaccional de los despliegues (funciones de ayuda, descargas web, sanitización de entorno, validaciones de salud y ahora el motor de desinstalación) se concentra en un único archivo: `lib/lib.sh`.
La propuesta sugiere renombrar/extraer la lógica de instalación hacia un `lib/install.sh` e independizar el motor de eliminación a `lib/uninstall.sh`.

---

## ⚖️ Análisis de Pros y Contras

### ✅ Pros (Ventajas de separar los ficheros)
1. **Separación de Responsabilidades (SoC)**: Mantiene el código altamente estructurado. Quien modifique cómo se instalan servicios tocará `install.sh`, quien arregle la eliminación tocará `uninstall.sh`.
2. **Previene el Anti-Patrón "God Object"**: A medida que implementemos la Fase 5 (Snapshots & Rollbacks) y utilidades avanzadas, el `lib.sh` actual podría acercarse a las 1000 líneas. Separarlo fomenta un diseño modular.
3. **Escalabilidad de Módulos**: Facilita agregar nuevos motores en el futuro (ej. `lib/metrics.sh` o `lib/rollback.sh`).

### ❌ Contras (Desafíos y Bloqueos en Shell Scripting)
1. **La pesadilla del "curl | bash" (El Bloqueo Principal)**: 
   Actualmente, el comando de un solo clic que usa el usuario ejecuta los scripts así:
   `curl ... caddy.sh | bash`
   
   Dentro de `caddy.sh`, para obtener los súper-poderes de la librería desde GitHub sin tenerla descargada, se hace:
   `source <(curl -fsSL "$_LIB_URL")`
   
   **Si dividimos `lib.sh` en varios archivos (`core.sh`, `install.sh`, `uninstall.sh`)**, obligaremos a que cada script de servicio ejecute **múltiples llamadas a curl** consecutivamente. Esto ralentiza significativamente el arranque del script e incrementa el riesgo de que una falla de red corrupta la ejecución del script a medio camino.

2. **Funciones Múltiplemente Compartidas (Core Dependencies)**: 
   Inclave al desinstalar, necesitamos las utilidades base como `log()`, `err()`, `root_protection()` y la comprobación de variables de entorno. Renombrar `lib.sh` enteramente a `install.sh` rompería la desinstalación a menos que extraigamos una *tercera* librería: `core.sh`. De lo contrario, tendríamos dependencias circulares complejas.

---

## 🛡️ Mitigación de los Contras: El Modelo Híbrido "Compilado"

Para abordar directamente los contras mencionados (especialmente el problema de las descargas múltiples por HTTP y las dependencias compartidas), la solución óptima es un entorno de desarrollo modular con un paso de despliegue ("build"):

### Arquitectura Propuesta:
1. En el repositorio crearemos una carpeta `lib/src/` con módulos temáticos:
   - `lib/src/01_core.sh` (Funciones de logs, guardias de root, detecciones base).
   - `lib/src/02_install.sh` (Lógicas de directorios, descargas, permisos).
   - `lib/src/03_health.sh` (Comprobaciones HTTP y de contenedores podman).
   - `lib/src/04_uninstall.sh` (El motor centralizado de eliminación).
   
2. **El "Compilador" de Librerías (`build.sh`)**:
   Implementaremos un pequeño script en Bash en el repositorio base. Este unirá en orden numérico (`cat lib/src/*.sh > lib/lib.sh`) todos los módulos en un único fichero monolítico de solo lectura (`lib.sh`).

---

## 💸 Análisis de la Nueva Deuda Técnica (Trade-offs)

Adoptar este modelo de compilación no es gratuito, pero sus cargas operativas pueden mitigarse siguiendo el siguiente esquema:

1. **Paso Obligatorio de Build**:
   - *El Problema*: Cualquier cambio manual realizado directamente en `lib/lib.sh` se perderá (será sobreescrito) la próxima vez que se ejecute el compilador.
   - *Deuda*: Requiere disciplina extrema para evitar editar el archivo de salida en lugar de los fuentes en `src/`.
   - *Solución*: El script compilador inyectará un encabezado de advertencia gigante en la primera línea de `lib/lib.sh`: `### DO NOT EDIT - AUTOGENERATED FROM lib/src/ ###`.

2. **Dificultad de Depuración (Debugging)**:
   - *El Problema*: Cuando un error de Bash indique "Fallo en la línea 483 de lib.sh", el número de línea no coincidirá con el archivo fuente original.
   - *Deuda*: Añade fricción cognitiva al rastrear fallos sintácticos.
   - *Solución*: Inyectar marcadores visuales (`# MODULE: 03_health.sh`) entre cada bloque de código durante la compilación. El desarrollador solo tendrá que hacer scroll up unos segundos desde la línea del error para identificar el módulo afectado.

3. **Carga Operativa Local**:
   - *El Problema*: Es necesario correr `./build.sh` antes de cada `git commit`. Si un desarrollador lo olvida, el repositorio tendrá código fuente desincronizado con el ejecutable de producción.
   - *Deuda*: Riesgo de desincronización entre desarrollo y despliegues reales.
   - *Solución*: Implementar un Git Hook (`pre-commit`) que ejecute `./build.sh` automáticamente. Si hay cambios en `lib/src/`, el hook compilará y añadirá el nuevo `lib/lib.sh` al commit de forma transparente.

### Veredicto:
Con estas mitigaciones automatizadas, los pros del aislamiento de código superan con creces a la deuda técnica. Tenemos un stack de grado corporativo 100% libre de fallos por error humano al compilar.
